3. **Не выдумывай** — если чего-то нет в CLAUDE.md или этой инструкции, спроси
4. **Не ломай существующее** — перед изменением проверь, что уже работает

### Задача 4: Wizard Engine (бэкенд)
```
Зависимости: Задача 2
Приоритет: блокирующая
```

**Что сделать:**
1. Возьми конфиг из `src/lib/engine/wizard-config.ts` — он уже готов
2. Создай `src/lib/engine/wizard-engine.ts` — движок:
   - `startSession(userId, briefType, templateId?)` → создаёт project + wizard_session + 6 wizard_steps (status: pending)
   - `getSessionState(sessionId)` → текущий шаг, статусы, structured_facts, context_summary
   - `processStep(sessionId, stepNumber, userAnswer)` → вызывает AI Service, обновляет данные шага, возвращает ответ
   - `overrideWarning(sessionId, warningId, description)` → добавляет в structured_facts.user_overrides[]
   - `checkSkipCondition(stepConfig, structuredFacts)` → возвращает true если шаг пропускается
   - `isStepComplete(stepNumber, extractedData, validationRules)` → проверяет валидацию
3. Автосохранение:
   - После каждого ответа: обновление wizard_steps в PostgreSQL
   - context_summary и structured_facts: обновление в wizard_sessions
   - Кэш текущего состояния: Redis (ключ `wizard:session:{id}`, TTL 1 час)
4. Проверка противоречий: после шагов 4 и 6 вызывает `checkContradictions()` из AI Service

**Критерии приёмки:**
- startSession создаёт все записи в БД
- processStep корректно вызывает AI Service с нужным промптом
- structured_facts обновляются и НЕ ТЕРЯЮТСЯ между шагами
- context_summary обновляется после каждого завершённого шага
- skip_condition работает (бот → пропуск шага адаптивности)
- Кэш Redis обновляется при каждом изменении состояния
- Коммит: `feat: Wizard Engine — движок шагов с кэшированием`

   - Логирование: каждый вызов Claude → запись в лог (session_id, step, input_tokens, output_tokens, latency_ms)

   - `retry_count` в wizard_steps

4. Подключи FeatureCatalog:
   - На шагах 3 и 4 передавай в промпт roles и features из feature_catalogs (по project_type, определённому на шаге 1)
5. Суммаризация: после каждого завершённого шага вызывай `summarizeContext()` и сохраняй результат

**Критерии приёмки:**
- Claude API вызывается со стримингом
- При 429/529 — retry с backoff (не crash)
- Каждый вызов логируется с токенами и latency
- FeatureCatalog подгружается на шагах 3-4
- Счётчик «3 попытки» работает: на 3-й → open_question
- Суммаризация генерирует context_summary 500-1000 токенов
- Коммит: `feat: AI Service — стриминг, retry, FeatureCatalog, суммаризация`

   ```
   src/app/api/
   ├── auth/[...nextauth]/route.ts     — NextAuth
   ├── wizard/
   │   ├── sessions/
   │   │   ├── route.ts                — POST: создать сессию
   │   │   └── [id]/
   │   │       ├── route.ts            — GET: состояние сессии
   │   │       ├── steps/
   │   │       │   └── [step]/
   │   │       │       └── route.ts    — PUT: отправить ответ
   │   │       ├── override/
   │   │       │   └── route.ts        — POST: игнорировать предупреждение
   │   │       └── generate/
   │   │           └── route.ts        — POST: сгенерировать ТЗ
   ├── documents/
   │   └── [id]/
   │       └── export/
   │           └── [format]/

   └── analytics/
   ```

**Критерии приёмки:**
- Все 6 эндпоинтов работают и возвращают корректные ответы
- Валидация zod отклоняет невалидные данные

- Коммит: `feat: API-роуты wizard + documents + analytics`

### Задача 16: Обработка ошибок и edge cases
```
Зависимости: Задачи 6, 9
Приоритет: средняя
```

**Что сделать:**
1. Глобальный error boundary: `src/app/error.tsx` — красивая страница ошибки с кнопкой «Попробовать снова»
2. AI timeout: если Claude не ответил за 30с → сообщение «AI думает дольше обычного. Попробуйте ещё раз.» + retry-кнопка
3. Потеря сессии: если wizard_session не найдена → redirect на дашборд с уведомлением
4. Пустой FeatureCatalog: если для project_type нет каталога → AI работает без каталога (fallback на генерацию из промпта)
5. Размытый ответ: если пользователь вводит <20 символов → inline-подсказка «Расскажите подробнее»
6. 429 Rate limit: показать пользователю «Слишком много запросов. Подождите минуту.»
7. Offline: если нет сети → уведомление + данные сохраняются в localStorage

**Критерии приёмки:**
- Ни один сценарий ошибки не крашит приложение
- Пользователь всегда видит понятное сообщение
- Данные не теряются при ошибках
- Коммит: `feat: обработка ошибок + edge cases + offline fallback`

## КРИТЕРИЙ ГОТОВНОСТИ MVP

MVP считается готовым, когда:

- [ ] Новый пользователь может пройти демо шага 1 без регистрации
- [ ] После регистрации — пройти все 6 шагов wizard с AI-диалогом
- [ ] AI задаёт уточняющие вопросы (не принимает первый ответ)
- [ ] AI предлагает фичи из FeatureCatalog на шаге 4
- [ ] AI ловит противоречия после шагов 4 и 6
- [ ] Пользователь может проигнорировать предупреждение (override)
- [ ] Если AI не получил ответ за 3 попытки — фиксирует open_question
- [ ] Сгенерированное ТЗ содержит все 7 разделов + открытые вопросы + полноту
- [ ] ТЗ экспортируется в Markdown и PDF
- [ ] Данные не теряются при перезагрузке страницы
- [ ] 15 AI-eval сценариев проходят с оценкой ≥ 4.0
- [ ] 14 аналитических событий записываются
- [ ] Работает на мобильном (375px)
